# 학교 시간표 관리 시스템 아키텍처 설문

> 기반 문서: SPEC.md
> 작성일: 2026-02-13

## 1. 프로젝트 맥락
- 아키텍처 방법론은 무엇으로 선택할 것인가?: **FSD(Feature-Sliced Design)** 를 채택한다. 제약 기반 생성/검증/부분 재계산처럼 복잡도가 높은 도메인을 장기적으로 유지보수하기 위해 레이어 규칙과 단방향 의존을 적용한다.
- 앱/렌더링 방식은 무엇으로 갈 것인가?: **TanStack Start** 기반으로 시작하며, 라우트 단위에서 SSR/CSR을 혼합해 사용한다.
- 초기 프로젝트 템플릿은 무엇으로 고정할 것인가?: `pnpm dlx shadcn@latest create --preset "https://ui.shadcn.com/init?base=base&style=nova&baseColor=neutral&theme=neutral&iconLibrary=hugeicons&font=geist&menuAccent=subtle&menuColor=default&radius=default&template=start&rtl=false" --template start` 명령으로 생성한 **고정 프리셋(Nova/Neutral + TanStack Start)** 을 기준으로 한다.
- UI/스타일링 원칙은 무엇인가?: **Tailwind CSS + 시멘틱 토큰 + 공통 UI 재사용**을 기본 원칙으로 고정한다.
- 핵심 비즈니스 가치는 무엇인가?: 자동 생성 정확도(필수 제약 100% 준수), 부분 수정 안정성(잠금 보존), 학기 중 교체 안전성(유효 후보만 노출)을 최우선 가치로 둔다.

## 2. 도메인 분석
- 핵심 엔터티는 무엇인가?: 학년, 반, 요일/교시, 교사, 과목, 시간표 셀, 제약 정책(연강/일일 제한), 잠금 상태, 변경 이력 이벤트를 핵심 엔터티로 본다.
- 주요 기능 시나리오는 무엇인가?: (1) 학기 시작 전 기초 시간표 자동 생성, (2) 생성 후 수동 수정 + 잠금 기반 부분 재계산, (3) 학기 중 교체 후보 탐색 및 확정 시나리오를 중심으로 설계한다.
- 유비쿼터스 언어는 어떻게 통일할 것인가?: `BASE`, `TEMP_MODIFIED`, `CONFIRMED_MODIFIED`, `LOCKED`, `연강` 용어를 공식 용어로 사용한다.

## 3. 비기능 요구사항
- 학생 연강 기본 정책은 어떻게 설정할 것인가?: **학년별/과목별 차등 정책(C)** 으로 운영한다. 공통 단일값 대신 정책 테이블에서 학년/과목 단위로 제한값을 관리한다.
- 교사 일일 최대 시수와 예외 승인 방식은?: **기본 5시수(B)**, 초과는 관리자 승인 프로세스로 처리한다.
- 변경 이력의 주차 경계는 어떻게 산정할 것인가?: **월요일 00:00(A)** 기준으로 주차를 구분한다.
- “확정” 권한은 누구에게 부여할 것인가?: **교무부 + 관리자 공동(B)** 으로 부여한다.
- 접근성/보안 원칙은 무엇인가?: 색상 외 텍스트·아이콘을 병행하고, 조회/수정/확정/정책 변경 권한을 분리하며 감사 가능한 이력 보존을 유지한다.

## 4. 아키텍처 방법론
- FSD를 어떻게 적용할 것인가?: `app`(조합/부트스트랩), `pages`(화면 단위), `widgets`(복합 UI), `features`(사용자 액션), `entities`(도메인 모델), `shared`(공통 유틸/인프라) 구조로 분리한다.
- 도메인 규칙과 UI 결합을 어떻게 통제할 것인가?: 제약 검증/탐색 로직은 `entities`/`features` 레이어에 두고, 화면 컴포넌트는 상태 표현과 사용자 상호작용에 집중한다.
- 부분 재계산/교체 탐색 같은 고복잡도 로직은 어디에 배치할 것인가?: 재사용 가능한 도메인 서비스 형태로 분리하고, UI에서는 유스케이스 단위 호출만 수행한다.

## 5. 데이터 및 상태 전략
- 서버 상태 관리는 무엇을 사용할 것인가?: **TanStack Query(A)** 를 사용해 생성/재계산/교체 후보 조회 등 서버 상태를 관리한다.
- URL 상태 전략은 무엇인가?: **적극 사용(A)** 하며, **TanStack Router** 검색 파라미터를 표준으로 사용한다. 필터, 선택된 학년/반, 주차, 보기 모드 등을 URL 상태로 관리해 공유 가능성과 복구성을 확보한다.
- 전역 클라이언트 상태는 어떻게 가져갈 것인가?: **최소화 + Zustand(A)** 를 채택한다. 전역 상태는 세션/선택 컨텍스트/일시 UI 상태로 제한한다.
- 로컬 UI 상태 원칙은 무엇인가?: 컴포넌트 코로케이션 원칙으로 가까운 위치에서 상태를 관리한다.
- 데이터 페칭 흐름은 무엇으로 볼 것인가?: CSR 기반의 `render-then-fetch`를 기본으로 하되, 사용자 체감 성능이 중요한 화면은 Query prefetch를 적용한다.

## 6. 테스트 및 품질
- 테스트 전략은 무엇으로 선택할 것인가?: **Vitest + RTL 중심(B)** 으로 구성하고, **브라우저 모드 테스트를 적극 활용**한다.
- E2E 범위는 어떻게 가져갈 것인가?: 핵심 사용자 경로(자동 생성, 잠금 기반 부분 재계산, 교체 후보 확정) 중심으로 최소 범위를 운영한다.
- 정적 분석 품질 게이트는 무엇인가?: ESLint/TypeScript 기반 검사와 FSD 레이어 의존 규칙을 CI 게이트로 적용한다.
- 회귀 리스크가 큰 영역은 어떻게 보호할 것인가?: 제약 검증 로직, 상태 전이(`BASE/TEMP/CONFIRMED`), 교체 후보 정렬 로직을 우선 테스트 대상으로 지정한다.

## 7. 추가 결정 사항
- 교체 후보 정렬 우선순위는 무엇인가?: **제약 위반 최소 > 기존안 유사도 > 공강 최소화(A)** 순으로 정렬한다.
