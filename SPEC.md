## 1. 개요 (Overview)

### 1.1 목표 (Goal)
제약 충족 기반 자동 생성과 부분 수정/교체를 지원하는 고등학교 시간표 운영 시스템을 구축한다.

### 1.2 배경 (Background)
학교 시간표는 교사·반·교시 제약이 복잡하고 학기 중 변경이 잦아 수작업 유지 비용이 크다. 본 시스템은 자동 생성, 연강 검증, 잠금 기반 부분 재계산, 변경 이력 시각화를 통해 운영 부담과 오류를 줄이는 것을 목표로 한다.

### 1.3 범위 (Scope)
#### 포함 사항 (In Scope)
- 학년(1~3), 반(학년당 1~15), 요일(월~금), 교시(기본 1~7, 설정 가능) 기반 시간표 모델
- 학교 운영 데이터(학교 구조, 교사, 과목, 시수, 고정 수업/출장) 입력·검증·영속화
- 교사 정보(담당 과목, 기준 시수, 반별 배정 시수, 선호/회피 조건) 관리
- 기초 시간표 자동 생성 (필수 제약 + 선호 제약)
- 생성 후 수동 편집(교사/과목 변경, 교시 이동) 및 셀 잠금
- 잠금 셀 보존 기반 부분 재계산
- 학기 중 교체 후보 탐색 및 자동 제약 검증
- 학생/교사 연강 제한 설정 및 검증
- 변경 이력 상태 관리 및 색상 시각화

#### 제외 사항 (Out of Scope)
- 출결/성적/생활기록부 등 타 학교 행정 시스템 전체 기능
- 모바일 앱 네이티브 구현
- AI 추천 고도화(대체 교사 추천, 피로도 모델링, 영향 분석 자동 리포트)의 1차 릴리스 포함

## 2. 사용자 및 시나리오 (Users & Scenarios)

### 2.1 대상 사용자
| 사용자 유형 | 설명 | 주요 니즈 |
|------------|------|----------|
| 시간표 담당 교무부 | 학기 시작 전 기초 시간표 편성 책임자 | 빠른 생성, 제약 위반 방지, 변경 추적 |
| 학년/과목 담당 교사 | 배치 선호/회피 조건을 제공하는 사용자 | 선호 반영, 과도한 연강 방지 |
| 관리자(교감/교무기획) | 최종 승인 및 운영 정책 결정 | 정책 기반 검증, 변경 이력 가시성 |

### 2.2 사용자 시나리오
#### 시나리오 1: 학기 시작 전 기초 시간표 생성
- **사용자**: 시간표 담당 교무부
- **상황**: 교사 정보와 시수 정보를 입력한 상태
- **행동**: 자동 생성 실행 및 결과 확인
- **기대 결과**: 필수 제약을 모두 충족한 시간표가 생성되고 선호 제약이 최대한 반영됨

#### 시나리오 2: 생성 후 부분 수동 수정
- **사용자**: 시간표 담당 교무부
- **상황**: 일부 교시 배치가 운영상 부적절
- **행동**: 특정 칸 변경 후 잠금 설정, 나머지 영역 재생성
- **기대 결과**: 잠금 셀은 유지되고 비잠금 영역만 재배치됨

#### 시나리오 3: 학기 중 교체 작업
- **사용자**: 시간표 담당 교무부
- **상황**: 출장/행사/운영상 사유로 교체 필요
- **행동**: 교체 대상 선택 후 후보 목록 확인 및 확정
- **기대 결과**: 충돌·연강·고정 조건을 만족하는 후보만 노출되어 안전하게 교체됨

## 3. 기능 요구사항 (Functional Requirements)

### 3.1 핵심 기능
| ID | 기능명 | 설명 | 우선순위 |
|----|--------|------|----------|
| F0 | 학교 운영 데이터 관리 | 학교 구조·교사·과목·시수·고정 수업/출장 데이터 입력 및 관리 | Must |
| F1 | 기초 시간표 자동 생성 | 필수/선호 제약을 반영해 초기 시간표 생성 | Must |
| F2 | 교사 배치 조건 관리 | 교사별 회피/선호/연강 허용 범위 설정 | Must |

| F5 | 학기 중 교체 후보 탐색 | 교체 가능 후보를 자동 검증 후 제공 | Must |
| F6 | 연강/일일 제한 검증 | 학생/교사 기준 연속·일일 제한 검증 | Must |
| F7 | 변경 이력 시각화 | 주차별 변경 상태 및 색상 표시 | Should |

### 3.2 기능 상세
#### F0: 학교 운영 데이터 관리
- **입력**: 학교 구조(학년 수, 학년별 반 수, 운영 요일, 일일 교시 수), 교사 정보(이름, 담당 과목, 기준 시수, 반별 배정 시수), 과목 목록, 고정 수업(특정 셀에 고정된 수업), 출장/행사 일정
- **처리**: 입력값 유효성 검증(시수 합계 정합성, 교사-과목 배정 누락 체크, 필수 필드 검증), IndexedDB 영속화
- **출력**: 생성(F1)·정책(F2)·재계산(F4)·교체(F5) 등 후속 기능에서 참조 가능한 운영 데이터 세트
- **예외 처리**: 시수 합계가 가용 교시를 초과하거나, 교사-과목 미배정 등 데이터 무결성 위반 시 저장 차단 및 수정 안내

#### F1: 기초 시간표 자동 생성
- **입력**: 학년/반/요일/교시 구조, 교사·과목·시수 정보, 고정 수업 정보, 제약 설정값
- **처리**: 필수 제약 충족을 우선 보장하고, 선호 제약을 점수화해 최적 배치 탐색
- **출력**: 각 학급/교사 기준 시간표와 제약 충족 결과
- **예외 처리**: 충족 불가능 시 원인 제약 목록과 우선 완화 후보를 안내

#### F2: 교사 배치 조건 관리
- **입력**: 교사별 회피 요일/교시, 선호 시간대(오전/오후), 연강 허용 한도
- **처리**: 조건 유효성 검증 및 정책 저장
- **출력**: 생성·재계산 시 반영 가능한 교사별 조건 세트
- **예외 처리**: 상충 조건(예: 전체 회피) 입력 시 저장 차단 및 수정 안내

#### F3: 수동 수정 및 잠금
- **입력**: 교시 셀 선택, 교사/과목 변경 또는 이동 명령, 잠금 여부
- **처리**: 즉시 충돌 검사 후 적용, 잠금 상태 기록
- **출력**: 변경 반영된 그리드 및 상태 태그(TEMP_MODIFIED/LOCKED)
- **예외 처리**: 충돌 또는 고정 셀 위반 시 변경 거부 및 사유 표시

#### F4: 부분 재계산
- **입력**: 현재 시간표, 잠금 셀 집합, 재계산 대상 범위
- **처리**: 잠금 셀 고정 후 비잠금 영역만 대상으로 재배치 수행
- **출력**: 잠금 보존된 갱신 시간표
- **예외 처리**: 재계산 실패 시 유지 가능한 최선안과 실패 원인 표시

#### F5: 학기 중 교체 후보 탐색
- **입력**: 교체 대상 셀(들), 교체 범위, 운영일자
- **처리**: 교사/반 충돌, 고정/출장, 학생·교사 연강 및 일일 제한 검증
- **출력**: 조건 만족 후보 목록(정렬 기준 포함)
- **예외 처리**: 후보 없음 시 제약 완화 시뮬레이션 결과 제시

#### F6: 연강/일일 제한 검증
- **입력**: 학교별 제한값(예: 학생 동일과목 최대 연속, 교사 최대 연속, 교사 일일 최대 시수)
- **처리**: 생성/수정/교체 시 실시간 또는 확정 전 검증
- **출력**: 위반 여부, 위반 위치, 위반 유형
- **예외 처리**: 정책 미설정 시 기본값 적용 또는 저장 차단(정책 결정 필요)

#### F7: 변경 이력 시각화
- **입력**: 변경 이벤트(시점, 사용자, 변경 전/후, 주차 태그)
- **처리**: 상태 전이(BASE → TEMP_MODIFIED → CONFIRMED_MODIFIED 등) 반영
- **출력**: 색상 규칙 기반 UI 표시(기초/임시/주차별 변경/잠금)
- **예외 처리**: 이력 누락 시 최근 정상 스냅샷 기준 복구 표시

## 4. 비기능 요구사항 (Non-Functional Requirements)

### 4.1 성능
- 기초 시간표 생성은 일반 학교 규모(학년 3, 반 최대 45, 주 5일, 7교시 기준)에서 실무 허용 시간 내 완료되어야 한다.
- 단일 셀 수정 시 충돌 검증 결과는 사용자 상호작용 흐름을 방해하지 않는 속도로 제공되어야 한다.
- 부분 재계산은 전체 재생성 대비 유의미한 시간 단축을 제공해야 한다.

### 4.2 접근성
- 색상 외에 아이콘/텍스트로 상태를 구분해 색각 이상 사용자도 변경 상태를 인지할 수 있어야 한다.
- 키보드 중심 편집(선택/이동/확정)이 가능해야 한다.

### 4.3 데이터 보존
- 변경 이력은 감사 가능한 형태로 보존되어야 한다.

### 4.4 호환성
- 최신 데스크톱 브라우저 환경에서 동일한 핵심 기능을 제공해야 한다.

## 5. 제약 조건 (Constraints)

### 5.1 기술적 제약
- 제약 충족 기반의 탐색/최적화가 가능한 엔진 구조가 필요하다.
- 필수 제약 위반은 어떤 경우에도 허용하지 않는다.
- 학교별 정책값(연강/일일 제한)을 런타임 설정으로 변경 가능해야 한다.
- 프론트엔드 프로젝트는 TanStack Start 기반(shadcn preset start template)으로 생성한다.
- 스타일링은 Tailwind CSS를 표준으로 사용한다.
- UI 스타일 값은 시멘틱 토큰을 통해 관리하고, 공통 UI 컴포넌트 재사용을 우선한다.

### 5.2 비즈니스 제약
- 학기 중 빈번한 변경을 지원해야 하므로 재편성 과정에서 기존 확정 시간표의 안정성이 중요하다.
- 운영 정책 변경 시 기존 시간표 데이터의 추적 가능성이 유지되어야 한다.

### 5.3 의존성
- 교사 기본 정보 및 주당 시수 데이터의 사전 입력
- 고정 수업/출장 등 운영 이벤트 데이터의 적시 반영

## 6. 성공 기준 (Success Criteria)

### 6.1 기능적 완료 조건
- [ ] 학교 운영 데이터(구조, 교사, 과목, 시수, 고정 수업)를 입력·저장하고 재실행 후 동일하게 복원할 수 있다.
- [ ] 자동 생성 결과가 필수 제약을 100% 충족한다.
- [ ] 교사 선호/회피 조건이 정책대로 반영된다.
- [ ] 잠금 셀 유지 상태에서 부분 재계산이 가능하다.
- [ ] 학기 중 교체 시 유효 후보만 제시된다.

### 6.2 품질 기준
- [ ] 변경 이력의 상태/색상 표기가 일관되다.
- [ ] 주요 사용자 시나리오(생성, 수정, 교체)가 중단 없이 수행된다.
- [ ] 제약 위반 시 원인과 조치 안내가 명확하다.

### 6.3 인수 기준 (Acceptance Criteria)
- [ ] 샘플 학교 데이터로 학기 시작용 기초 시간표를 생성할 수 있다.
- [ ] 임의의 잠금 셀을 포함한 부분 재계산이 정상 동작한다.
- [ ] 학생/교사 연강 제한 위반 케이스를 시스템이 검출한다.
- [ ] 주차별 변경 이력이 시간표/교사표/학급표에 동기화 반영된다.

## 7. 경계 조건 및 엣지 케이스 (Edge Cases)

| 상황 | 예상 동작 | 비고 |
|------|----------|------|
| 특정 교사의 회피 조건이 과도하여 배치 불가 | 저장 또는 생성 단계에서 충돌 원인 안내 | 조건 완화 가이드 필요 |
| 잠금 셀이 과도해 재계산 불가 | 실패 사유와 최소 해제 필요 셀 제안 | 운영 지원 기능 |
| 교체 후보가 0개 | 후보 없음 안내 + 완화 가능한 제약 옵션 표시 | 확정 전 단계 |
| 학생 연강과 교사 연강 제약이 상충 | 학교 정책 우선순위에 따라 검증 결과 제시 | 정책 정의 필요 |
| 교시 수 변경(예: 7→6) | 영향 받는 배치 목록과 재생성 필요 범위 안내 | 학기 단위 설정 권장 |
| 교사 시수 합계가 가용 교시를 초과 | 입력 단계에서 초과 경고 및 저장 차단 | 시수 배분 재조정 안내 |
| 과목에 배정된 교사가 없음 | 저장 시 누락 과목-교사 배정 안내 | 생성(F1) 전 필수 해소 |

## 8. 용어 정의 (Glossary)

| 용어 | 정의 | 비고 |
|------|------|------|
| 기초 시간표(BASE) | 자동 생성된 초기 확정 전 시간표 | 기준 스냅샷 |
| 임시 수정(TEMP_MODIFIED) | 사용자가 편집했으나 최종 확정 전 상태 | 색상 표시 대상 |
| 확정 수정(CONFIRMED_MODIFIED) | 운영 승인 후 반영된 변경 상태 | 주차 태그 포함 |
| 잠금(LOCKED) | 재계산 시 변경되지 않도록 보호된 셀 | 아이콘 표시 |
| 연강 | 동일 대상(학생/교사)이 연속 교시에 수업하는 상태 | 정책으로 제한 |
